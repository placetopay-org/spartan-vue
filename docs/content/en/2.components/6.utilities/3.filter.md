# Usage

`SFilter` lets users add, configure, and apply filters on a typed set of fields. The component is fully controlled: the `filters` prop holds the **read-only configuration**, and a `v-model` holds the **applied filter values**.

::component-preview{file="SFilter/basic"}
::

The `filters` prop is a record of field id → field definition. Each definition picks one of seven `type` values: `text`, `number`, `amount`, `date`, `dateRange`, `options`, `selection`. TypeScript only allows the operators that make sense for the chosen `type`.

The `v-model` value is `Record<string, { operator, value }>`. Only the filters the user has applied appear in the model. `delete value.foo` removes a filter; `value = {}` clears every filter.

## Field types

### text

Single-line text input. The default operator is `contains`.

::component-preview{file="SFilter/one-input"}
::

### number / amount / date

`number` and `date` render single inputs with type-appropriate UI; `amount` adds a currency selector. They share the same default of one input per operator (two when the operator is `between` or `notBetween`).

### dateRange

A dedicated date-range picker. The default operator is `between`.

::component-preview{file="SFilter/two-inputs"}
::

### options

A discrete choice list. `choices` can be either `string`s (id and label are the same) or `{ id, label }` objects. Set `multiple: true` to allow several selections.

::component-preview{file="SFilter/options"}
::

### selection

A free-form tag input — the user types their own values.

::component-preview{file="SFilter/selection"}
::

## Existence operators

Every field type supports `exist` / `notExist`. These render no value input.

::component-preview{file="SFilter/none"}
::

## Custom operators

Custom operators live in `customOperators` (separate from the predefined `operators` array). Each custom operator has:

- `id` — unique operator id
- `label` — text shown in the operator dropdown
- `inputs` — optional `0`, `1`, or `2`. Defaults to the field type's natural count (`1` for everything except `dateRange`, which is `2`). Use `inputs: 0` to mimic existence-style behavior.
- `tag` — optional badge formatter. A string renders as-is, a function receives the current value and returns the rendered tag, an absent `tag` falls back to the raw value.

::component-preview{file="SFilter/custom-operators"}
::

## Initial state

Initialize the `v-model` ref with pre-applied filters. The component renders the badges and excludes those fields from the "Add filter" list. Useful for query-param-driven UIs, saved presets, and rehydration from localStorage or a backend.

::component-preview{file="SFilter/initial-state"}

## Permanent fields

Set `permanent: true` to prevent a field from being cleared or removed by the user. The badge keeps its content but no remove control is rendered; `clear` leaves permanent entries intact.

## i18n / reactive labels

Predefined operator labels are translated automatically through the `$spartan.filter.operator.*` namespace — when the locale changes, `vue-i18n` re-renders the badges on its own.

Everything else you write in the `filters` object — `label`, `choices`, and `customOperators` labels/tags — is a **literal string**. If you hardcode them, they will not react to a locale change. To make them locale-aware, wrap `filters` in a `computed` that depends on `vue-i18n`'s `t`:

```ts
const { t } = useI18n();

const filters = computed(() => ({
    status: {
        type: 'options',
        label: t('myApp.statusLabel'),
        choices: [{ id: 'active', label: t('myApp.statusActive') }],
        operators: ['equal'],
    },
}));
```

When the locale switches, `t()` recomputes, `filters` returns a new object, and the badges re-render with the new labels.

::component-preview{file="SFilter/i18n"}
::

## Saved filters

Provide a `saved` prop to render the saved-filters panel. The component never mutates the array — persistence is the consumer's responsibility. Listen to:

- `@save="(name, snapshot) => …"` — push to your store.
- `@load="(snapshot) => …"` — the `v-model` is already updated by the time this fires.
- `@delete="(name) => …"` — remove from your store.

::component-preview{file="SFilter/saved"}

## Validation

Each field can declare a `validate` function. It receives `(value, operator)` and returns an error message string (`null` if valid). Async validation is supported.

```vue
<script setup lang="ts">
import { ref } from 'vue';
import { SFilter, type SFilterField, type SFilterValue } from '@placetopay/spartan-vue';

const filters = {
    bin: {
        type: 'text',
        label: 'Card BIN',
        operators: ['equal'],
        validate: async (value: any) => (/^\d{6}$/.test(value) ? null : 'BIN must be 6 digits'),
    },
    amount: {
        type: 'amount',
        label: 'Amount',
        currency: 'USD',
        operators: ['between'],
        validate: (value: any, operator: string) =>
            operator === 'between' && parseFloat(value[0]) > parseFloat(value[1])
                ? 'Min value must be less than max value'
                : null,
    },
} satisfies Record<string, SFilterField>;

const value = ref<SFilterValue>({});
</script>

<template>
    <SFilter v-model="value" :filters="filters" />
</template>
```

## Operators reference

`OPERATORS_BY_TYPE` is exported publicly so you can build custom UIs against the same set.

```ts
import { OPERATORS_BY_TYPE } from '@placetopay/spartan-vue';

console.log(OPERATORS_BY_TYPE.text); // ['contains', 'notContains', ...]
```

| Field type    | Allowed operators                                                                                                                  |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `text`        | `contains`, `notContains`, `startsWith`, `endsWith`, `equal`, `notEqual`, `exist`, `notExist`                                       |
| `number`      | `equal`, `notEqual`, `greaterThan`, `greaterThanOrEqual`, `lessThan`, `lessThanOrEqual`, `between`, `notBetween`, `exist`, `notExist` |
| `amount`      | same as `number`                                                                                                                   |
| `date`        | `equal`, `before`, `after`, `today`, `yesterday`, `lastWeek`, `lastMonth`, `lastYear`, `exist`, `notExist`                          |
| `dateRange`   | `between`, `notBetween`, `lastWeek`, `lastMonth`, `lastYear`, `exist`, `notExist`                                                  |
| `options`     | `equal`, `notEqual`, `exist`, `notExist`                                                                                           |
| `selection`   | `equal`, `notEqual`, `exist`, `notExist`                                                                                           |

## Exposed methods

The component exposes `apply` and `clear` via template ref.

```vue
<template>
    <SFilter ref="filterRef" v-model="value" :filters="filters" />
    <button @click="filterRef?.apply()">Apply</button>
    <button @click="filterRef?.clear()">Clear</button>
</template>
```

## Props

| Prop               | Type                              | Default | Description                                                                                          |
| ------------------ | --------------------------------- | ------- | ---------------------------------------------------------------------------------------------------- |
| `filters`          | `Record<string, SFilterField>`    | —       | Read-only field definitions, keyed by id.                                                            |
| `modelValue`       | `SFilterValue`                    | `{}`    | The `v-model` of applied filters: `Record<string, { operator, value }>`.                             |
| `saved`            | `SFilterSaved[]`                  | —       | Optional saved-filter entries. Renders the saved panel when defined (even when empty).               |
| `hideApplyButton`  | `boolean`                         | `false` | Hide the Apply button.                                                                               |
| `hideClearButton`  | `boolean`                         | `false` | Hide the Clear button.                                                                               |
| `applyWhenClear`   | `boolean`                         | `false` | When clearing, also emit `apply` with the new (post-clear) value.                                    |
| `immediateApply`   | `boolean`                         | `false` | Emit `apply` on mount with the initial `modelValue`.                                                 |
| `responsive`       | `boolean`                         | `true`  | Whether internal popovers adopt the responsive layout.                                               |

## Events

| Event                    | Payload                              | Description                                                                                                                |
| ------------------------ | ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `update:modelValue`      | `SFilterValue`                       | Standard `v-model` plumbing.                                                                                                |
| `apply`                  | `SFilterValue`                       | The user clicked Apply (or `immediateApply` fired on mount).                                                                |
| `clear`                  | `SFilterValue`                       | The user clicked Clear. Payload is the post-clear value (permanent fields preserved).                                       |
| `save`                   | `name: string, snapshot: SFilterValue` | The user saved the current value under `name`. Persist as you wish.                                                         |
| `load`                   | `snapshot: SFilterValue`             | A saved entry was loaded. The `v-model` is already updated; this fires after.                                               |
| `delete`                 | `name: string`                       | The user deleted a saved entry. Update your store accordingly.                                                              |

## Migration from v1

The v2 API is a clean break from v1. There is no runtime adapter. Below are the six canonical migration scenarios.

### 1. Simple text filter

```ts
// BEFORE
const fields = ref<TField[]>([
    {
        id: 'name',
        name: 'Nombre',
        interfaces: { oneInput: { type: 'text', operators: ['contains'] } },
    },
]);

function onApply(applied: TField[]) {
    const used = applied.filter((f) => f.state);
    refetch(used);
}

<SFilter :fields="fields" @apply="onApply" />
```

```ts
// AFTER
import type { SFilterValue } from '@placetopay/spartan-vue';

const filters = {
    name: { type: 'text', label: 'Nombre', operators: ['contains'] },
} as const;

const value = ref<SFilterValue>({});

function onApply(v: SFilterValue) {
    refetch(v);
}

<SFilter v-model="value" :filters="filters" @apply="onApply" />
```

### 2. Multi-select with operators

```ts
// BEFORE
{
    id: 'brand',
    name: 'Brand',
    interfaces: {
        options: {
            multiple: true,
            operators: ['equal', 'notEqual', 'contains', 'notContains'],
            options: ['Nike', 'Adidas', { id: 'pma', label: 'Puma' }],
        },
    },
}
```

```ts
// AFTER
const filters = {
    brand: {
        type: 'options',
        label: 'Brand',
        multiple: true,
        operators: ['equal', 'notEqual'],
        choices: ['Nike', 'Adidas', { id: 'pma', label: 'Puma' }],
    },
} as const;
```

`contains` / `notContains` are no longer valid on an `options` field — TypeScript catches it at compile time. If you need free-text matching on the same column, declare a separate `text` filter.

### 3. Custom operator

```ts
// BEFORE
{
    id: 'price',
    name: 'Price',
    interfaces: {
        oneInput: {
            type: 'amount',
            currency: 'USD',
            operators: [
                'equal',
                { id: 'thousands', label: 'Miles', tag: (v) => `${v}K` },
            ],
        },
    },
}
```

```ts
// AFTER
const filters = {
    price: {
        type: 'amount',
        label: 'Price',
        currency: 'USD',
        operators: ['equal'],
        customOperators: [{ id: 'thousands', label: 'Miles', tag: (v: any) => `${v}K` }],
    },
} as const;
```

### 4. Date range

```ts
// BEFORE
{
    id: 'createdAt',
    name: 'Created at',
    interfaces: {
        twoInputs: { type: 'date', operators: ['between', 'lastWeek'] },
    },
}
```

```ts
// AFTER
const filters = {
    createdAt: {
        type: 'dateRange',
        label: 'Created at',
        operators: ['between', 'lastWeek'],
    },
} as const;
```

There is no `twoInputs` interface anymore — `dateRange` is its own field type with a `dateRange`-specific operator union.

### 5. Saved filters

```ts
// BEFORE
const fields = ref([...]);
const saved = ref<TSaveData[]>([]);

<SFilter
    :fields="fields"
    :saved="saved"
    @save="(all) => saved.value = all"
    @load="(filters) => { /* component already mutated fields */ refetch(); }"
/>
```

```ts
// AFTER
import type { SFilterValue, SFilterSaved } from '@placetopay/spartan-vue';

const value = ref<SFilterValue>({});
const saved = ref<SFilterSaved[]>([]);

function onSave(name: string, snapshot: SFilterValue) {
    saved.value = [...saved.value, { name, snapshot }];
}
function onLoad(_snapshot: SFilterValue) {
    // value is already updated via v-model
    refetch(value.value);
}
function onDelete(name: string) {
    saved.value = saved.value.filter((s) => s.name !== name);
}

<SFilter
    v-model="value"
    :filters="filters"
    :saved="saved"
    @save="onSave"
    @load="onLoad"
    @delete="onDelete"
/>
```

If you have v1-shaped saved data persisted (e.g., in localStorage), run this one-time transformation when adopting v2:

```ts
function migrateSavedV1ToV2(v1: { name: string; filters: any[] }[]) {
    return v1.map(({ name, filters }) => ({
        name,
        snapshot: Object.fromEntries(
            filters
                .filter((f) => f.state)
                .map((f) => [f.id, { operator: f.state.operator, value: f.state.value }]),
        ),
    }));
}
```

### 6. Initial state / presets

```ts
// BEFORE — only achievable by mutating field definitions
const fields = ref<TField[]>([
    {
        id: 'status',
        name: 'Estado',
        interfaces: { options: { operators: ['equal'], options: ['active', 'paused'] } },
        state: { operator: 'equal', value: 'active' }, // ← pre-applied via mutation
    },
]);
```

```ts
// AFTER — initialize the v-model
const filters = {
    status: {
        type: 'options',
        label: 'Estado',
        choices: ['active', 'paused'],
        operators: ['equal'],
    },
} as const;

const value = ref<SFilterValue>({
    status: { operator: 'equal', value: 'active' }, // ← pre-applied via v-model
});

<SFilter v-model="value" :filters="filters" />
```

Works for query params, saved-from-backend filters, and localStorage rehydration.
